# 프로젝트 구조 및 기능 설명 가이드 (신입 개발자용)

안녕하세요! **Text Extractor** 프로젝트에 오신 것을 환영합니다. 이 프로젝트는 고정확도 OCR(광학 문자 인식) 기술을 활용하여 이미지에서 텍스트를 추출하고, 이를 한국어 문맥에 맞게 다듬어주는 웹 애플리케이션입니다.

프로젝트의 각 파일과 디렉토리가 어떤 역할을 하는지 핵심 위주로 설명해 드릴게요.

---

## 1. 루트 디렉토리 (Main Application)
가장 먼저 접하게 되는 메인 서비스 코드입니다. 정적 파일(HTML/JS/CSS) 기반으로 구성되어 있습니다.

*   **`index.html`**: 서비스의 메인 레이아웃입니다.
    *   이미지 업로드 영역, OCR 엔진 선택(Google Vision vs Tesseract), 결과 편집창으로 구성되어 있습니다.
    *   Google Cloud Vision API 키 설정을 브라우저 로컬 저장소(`localStorage`)에 보관하도록 설계되어 보안과 편의성을 챙겼습니다.
*   **`script.js`**: 애플리케이션의 모든 핵심 로직이 담긴 파일입니다. (약 500줄)
    *   **이미지 처리**: 드래그 앤 드롭 및 여러 장의 이미지를 순서 변경하는 로직이 포함되어 있습니다.
    *   **OCR 연동**: Google Vision API(REST fetch)와 Tesseract.js(로컬 분석)를 모두 지원합니다.
    *   **후처리 로직(`processBookText`)**: OCR 특유의 불필요한 줄바꿈을 제거하고, 한국어 조사(은/는/이/가 등)를 감지해 문장을 매끄럽게 이어주는 '스마트 결합' 기능이 핵심입니다.
    *   **맞춤법 교정**: 외부 API(`https://speller.town`)와 연동하여 추출된 텍스트의 오타를 수정합니다. 긴 텍스트는 청크(Chunk) 단위로 나누어 처리합니다.
*   **`style.css`**: 현대적이고 깔끔한 UI를 위한 스타일 시트입니다.
    *   변수(`--root`)를 사용하여 색상 시스템을 관리하며, 반응형 디자인과 부드러운 애니메이션이 적용되어 있습니다.

---

## 2. `opticon-ocr` 디렉토리 (Modularized Version)
루트의 `script.js`를 기능별로 쪼개어 관리하기 편하게 만든 **모듈화 버전**입니다.

*   **특징**: Vite 빌드 도구를 사용하며, 관심사 분리(SoC) 원칙에 따라 코드가 나누어져 있습니다.
*   **`src/js/` 주요 모듈**:
    *   `state.js`: 전역 상태 관리 (업로드된 파일 목록 등).
    *   `ocr.js`: OCR 엔진 호출 로직.
    *   `extract.js`: 텍스트 추출 프로세스 제어.
    *   `upload.js`: 파일 업로드 및 드래그 앤 드롭 UI.
    *   `imageList.js`: 이미지 목록 렌더링 및 순서 변경.
    *   `output.js`: 결과 텍스트 출력 및 복사/저장.

---

## 3. `speller-api-main` 디렉토리 (Backend Server)
한국어 맞춤법 교정을 담당하는 별도의 서버 프로젝트입니다.

*   **기술 스택**: Bun(Runtime) + TypeScript + Elysia(Web Framework).
*   **역할**: 클라이언트의 요청을 받아 한국어 맞춤법 검사기 엔진과 통신하고 결과를 반환합니다.
*   **주요 파일**:
    *   `server.ts`: API 엔드포인트를 정의하고 요청을 처리하는 메인 서버 코드.
    *   `struct.ts`: 데이터 구조(타입) 정의.

---

## 4. 핵심 비즈니스 로직 설명 (Tip)

새로 합류하신 개발자분이 특히 주목해야 할 부분은 **`script.js`의 후처리 로직**입니다.

1.  **줄바꿈 최적화**: 단순히 텍스트를 추출하면 책의 줄바꿈이 그대로 남습니다. 이를 문맥에 따라 문단으로 합치는 과정이 매우 중요합니다.
2.  **API 호출 최적화**: Google Vision API 호출 시 Base64 인코딩을 처리하는 방식과, 맞춤법 API 호출 시 서버 부하를 줄이기 위한 대기 로직(`429 Too Many Requests` 처리)을 확인해 보세요.
3.  **순서 제어**: 이미지 여러 장을 하나의 텍스트 파일로 합칠 때, 사용자가 UI에서 드래그로 순서를 바꾼 것이 결과물에 반영되도록 리스트 인덱스를 관리합니다.
